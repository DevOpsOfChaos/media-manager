# v0.3 Baseline Protocol — PySide6 + Multi-Source Organizer

## Goal of this iteration

Create the first testable desktop baseline that is aligned with the actual product direction:

- modern desktop GUI foundation
- English as default language
- multiple source folders
- one target folder
- current organizer behavior preserved

This iteration is intentionally limited. It does **not** try to solve duplicates, visual comparison, saved import sets, or localization switching yet.

## What changed

### UI stack

- Replaced the temporary Tkinter GUI baseline with a PySide6 desktop baseline.
- Added a darker, more modern desktop layout for the organizer flow.

### Organizer workflow

- Source folders are now managed as a list.
- Target folder remains a single destination.
- ExifTool path remains configurable.
- Preview / apply flow remains intact.
- Copy / move behavior remains intact.

### Core logic

- Sorting configuration now supports multiple source folders.
- File iteration deduplicates overlapping source-path input by resolved path.

### CLI

- CLI now uses repeatable `--source` flags plus an explicit `--target` flag.

### Documentation

- README, architecture notes, and roadmap were updated to reflect the real product direction.
- This protocol file was added to document the scope and expectations of the iteration.

## Local validation focus

The following checks matter for the next real test cycle:

1. install dependencies after pulling the new version
2. confirm that PySide6 starts correctly on the local Windows machine
3. add multiple source folders in the GUI
4. run a preview into one target folder
5. confirm that copy / move still behave correctly
6. confirm that ExifTool-based date resolution still works when ExifTool is available
7. confirm that the CLI still works with the new multi-source flags

## Known limitations of this baseline

- source-folder groups are not saved yet
- there is no rename workflow yet
- there is no duplicate workflow yet
- there is no image/video comparison view yet
- there is no language switcher yet
- localization architecture is still a future step, not a finished feature

## Why this scope was chosen

Because the project needed a **testable product baseline**, not more speculative features.

Trying to add import sets, duplicate handling, comparison UI, and localization in the same iteration would have been scope drift and would have reduced the chance of getting a stable testable build.
