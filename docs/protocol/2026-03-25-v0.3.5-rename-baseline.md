# v0.3.5 Protocol — Rename Baseline

## Goal of this iteration

Add the first real Rename workspace without drifting into a broader rule engine.

The focus of this iteration is:

- a dedicated Rename page in the app shell
- preview-first template-based renaming
- collision-safe rename planning
- in-place apply flow
- minimal but real usability

## What changed

### App shell

- Activated the **Rename** workspace on the home page.
- Added a dedicated Rename page in the navigation.

### Rename workflow

- A user can choose one or more source folders.
- The workspace scans media files recursively inside those folders.
- Filenames are rendered from a user template.
- The rename flow supports preview and apply.
- Renames happen in place, inside the existing folders.

### Rename template

The baseline template exposes these placeholders:

- `{year}`
- `{month}`
- `{day}`
- `{hour}`
- `{minute}`
- `{second}`
- `{stem}`
- `{suffix}`
- `{index}`

Default template:

`{year}{month}{day}_{hour}{minute}{second}_{stem}{suffix}`

### Safety behavior

- Preview is still the default.
- If the rendered name matches the current file name, the item is marked as unchanged.
- Name collisions are resolved safely with incrementing suffixes.

### Validation support

- Added tests for template rendering.
- Added tests for collision-safe rename planning.
- Added tests for apply behavior.

## What this iteration does not do

- no bulk rule editor yet
- no advanced regex rename system yet
- no duplicate-assisted rename logic yet
- no localization switcher yet

## Local validation focus

After pulling this version, validate the following:

1. the Rename page opens correctly
2. multiple source folders can be added
3. preview shows current and new names clearly
4. applying a rename changes filenames as expected
5. identical target names are deconflicted cleanly
6. unchanged items are reported correctly

## Why this scope was chosen

Because Rename is the next real product module, but a useful baseline matters more than a bloated first draft.

This iteration adds a real second workspace without pretending that the project already has a full rename rules engine.
