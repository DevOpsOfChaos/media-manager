# v0.3.4 Protocol — Organizer Import Sets

## Goal of this iteration

Reduce repetitive organizer setup work without drifting into new major product modules.

The focus of this iteration is:

- reusable import sets inside the existing organizer workflow
- compact UI controls instead of more page clutter
- persistence in the existing settings store
- test coverage for the import-set data helpers

## What changed

### Organizer workflow

- Added saved import sets to the organizer page.
- A user can now save the current organizer setup as a reusable set.
- A saved set stores:
  - source folders
  - target folder
  - target template

### Organizer UI

- Added compact import-set controls directly in the source-folder area.
- Import sets can be saved, loaded, and deleted.
- The implementation uses a compact combo-box based flow instead of opening another larger page.

### Settings layer

- The existing settings store now also persists import sets.
- Import sets are normalized and deduplicated before being stored.
- Case-insensitive replacement is supported when a set with the same name is saved again.

### Validation support

- Added tests for import-set normalization, update, lookup, and deletion.

## What this iteration does not do

- no rename workflow yet
- no duplicate workflow yet
- no compare workflow yet
- no localization switcher yet

## Local validation focus

After pulling this version, validate the following:

1. save an organizer setup as an import set
2. close and reopen the application
3. confirm that the import set still exists
4. load the import set and verify source folders, target folder, and template
5. overwrite an existing set name and confirm it updates cleanly
6. delete a set and confirm it disappears cleanly

## Why this scope was chosen

Because repeated manual organizer setup is wasted user time.

Import sets improve the existing workflow immediately, while staying inside the current product scope and without pretending that rename, duplicates, or compare are already implemented.
