# v0.3.1 Protocol — Organizer GUI Polish

## Goal of this iteration

Improve the quality of the already testable organizer baseline without expanding into new major feature areas.

The focus of this iteration is:

- better visual structure
- clearer run feedback
- better organizer ergonomics
- no scope drift into import sets, duplicates, or compare workflows

## What changed

### Organizer UI

- Reworked the window into a more structured dashboard-style layout.
- Added summary cards for source count, target state, mode, and run mode.
- Added an explicit action to open the target folder from the GUI.
- Added clearer result feedback above the result table.
- Kept the organizer workflow focused on the current scope instead of adding fake placeholder modules.

### Source-folder handling

- Source-folder list management was kept simple and direct.
- Repeated source-path input is still tolerated and deduplicated by the core iterator.

### Validation support

- Added a test covering repeated source-path input so the organizer logic is less likely to regress while the UI evolves.

## What this iteration does not do

- no saved import sets yet
- no rename workflow yet
- no duplicate workflow yet
- no comparison views yet
- no language switcher yet

## Local validation focus

After pulling this version, validate the following:

1. the GUI still starts normally
2. the summary cards update when source folders are added or removed
3. the target-folder open action works correctly
4. preview and apply still work correctly
5. result feedback is clearer than in the previous baseline

## Why this scope was chosen

Because once the baseline is testable, the next leverage point is usability.

Jumping straight into import sets or duplicate workflows before cleaning up the organizer UX would have increased complexity without improving the quality of the currently testable build.
